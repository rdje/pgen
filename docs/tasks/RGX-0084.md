## Goal

Fix `PGEN-RGX-0084` (downstream report
`/Users/richarddje/Documents/github/rgx/pgen-issues/PGEN-RGX-0084.yaml`,
artifact `artifacts/PGEN-RGX-0084/ast_dump.json`): the released
`regex` parser (`1.1.74`, contract `1.1.76`) classifies a two-digit
numeric backslash escape `\NN` as a `{kind:"numeric",
type:"backreference", index:N}` whenever the **whole pattern** has
≥ N capturing groups, instead of applying PCRE2's compile-time rule —
`\NN` is a back reference iff there are ≥ N capturing groups **opened
up to that parse position**; otherwise it is an octal escape (if it
qualifies). Reproducer:
`()()()()()()()()()(?:(?(10)\10a|b)(X|Y))+` — only 9 groups precede
`\10`; group 10 (`(X|Y)`) is opened *after*, so `\10` must be octal
`\010` (U+0008), not a backref index 10.

## Non-Goals

- Not changing single-digit `\1` behavior (the report confirms it is
  correct; the bug is exclusively the forward-reference two+-digit
  path where N exceeds groups-seen-so-far).
- Not the `\g{...}` / `\g<...>` subroutine/explicit-backref forms
  (those are explicit, not subject to the octal-disambiguation rule).
- Not an RGX-side workaround (doctrine: parsing issues are fixed in
  PGEN, not hacked around downstream).

## Acceptance Criteria

- **FAMILY fix (user-directed 2026-05-18), not just the reproducer.**
  The `.1` evidence proved PGEN has **no** `\NN` disambiguation at
  all: `backreference = "\\" /([1-9][0-9]*)/` unconditionally emits a
  numeric backref for *every* `\<nonzero-digit>…` (verified: `\10`,
  `\12`, `\99`, `\101`, `\8` all → numeric backref regardless of group
  count/position; only the separate `\0…` `octal_escape` path is
  correct). The fix implements PCRE2's **general** compile-time rule
  for the whole `\<digit>…` numeric-escape family, not a
  reproducer-specific patch.
- PCRE2's exact rule (pcre2pattern(3) BACKREFERENCES) is honored for
  the family: the **longest** digit run is read as integer N; it is a
  back reference iff there are **≥ N capturing groups opened up to
  that source position**; otherwise the digits are taken as an octal
  value **if they qualify** (one digit 0–7; or two digits 0–7 each
  (≤ 077); or up to three octal digits) — **re-splitting** the digit
  run into an `octal_escape` atom (≤ 3 leading octal digits) plus any
  remaining digits as literal-digit atoms (e.g. `\101`→octal 0o101;
  `\1234`→octal 0o123 then literal `4`); and `\8`/`\9` with no
  matching group and no octal qualification fall back to the
  literal-digit per PCRE2. Existing `\1`-with-group→backref and
  `\0…`→octal behavior is preserved.
- For the report's reproducer the `\10` token's AST atom becomes an
  octal-escape (its 1-byte expansion), **not** numeric backref 10.
- No regression: all existing regex parser/embedding-API/
  integration-contract tests + `regex_parser_book_gate` +
  `mdbook_docs_gate` green; the regex AST-shape contract holds.
- Released-parser bug remediated end to end: `REGEX-00NN` row in
  `docs/contracts/PGEN_RELEASED_PARSER_BUG_LEDGER.md` (downstream ref
  `PGEN-RGX-0084`), regex release/contract + schema bump per the
  per-family convention, book + contract + shape-contract same-commit
  lockstep, CHANGES/DEV/LIVE/memory.

## Open Questions

- Rust post-pass vs grammar-level? **RESOLVED `.1` (premise error
  retracted): grammar-level semantic annotations.** The "EBNF is
  context-free → must be a Rust post-pass" claim was wrong;
  `@predicate`/`@emit_fact`/`@open_scope` are always-on
  (`parser-hooks.md:211`) and `systemverilog.ebnf` proves the
  `@emit_fact`+`@predicate has_fact` idiom for parse-time, source
  -order, context-sensitive decisions. The regex book
  (`examples-escapes.md:112-121`) currently documents the
  unconditional-numeric-backref behavior with a now-**superseded**
  rationale ("PEG cannot express it; left to consumers via post-parse
  semantic analysis") — `.3` must correct that passage in lockstep.
- `.2` design detail — **RESOLVED (P2 design-lock, verified against
  `ast_based_generator.rs` codegen + the proven `systemverilog.ebnf`
  idiom; consult-annotation-docs discipline honored):**
  1. **Predicate phase = `post`, NOT `pre`** (the earlier "pre" guess
     was wrong). `ast_based_generator.rs:908-943` evaluates a `pre`
     predicate *before* the rule body parses → `$2`
     (`backreference_digits`) is **not yet bound** → predicate returns
     `None` → not blocked → fix would no-op. `:961-1024` evaluates a
     `post` predicate *after* the body parses → `$2` bound +
     `resolve_semantic_predicate_spec_against_content` resolves it →
     `Some(false)` → `Err(ParseError::Backtrack{position:
     node.span.start})` → the PEG retreats and tries the next
     alternative. This mirrors the proven `systemverilog.ebnf`
     `@predicate {…, phase: post}` idiom exactly.
  2. **Gate only the numeric branch via a named-rule lift** (the
     proven `pp_if_keyword` / op-rule lift idiom): new single-alt rule
     `numeric_backreference = "\\" backreference_digits -> {…kind:
     "numeric", index:$2}` carrying
     `@predicate:{name:fact_count_at_least,
     args:[regex_capture_group,$2], phase:post}`; `backreference`'s
     first alternative becomes `numeric_backreference -> $1`
     (passthrough; AST shape unchanged). A rule-level predicate keyed
     on `$2` must NOT sit on `backreference` itself — it would also
     gate the numeric `\g{N}`/`\g<N>` subroutine alternatives (whose
     `$n` is also digits) and corrupt them (those are explicit, never
     octal — a Non-Goal).
  3. **`@-`directive→rule binding = the immediately FOLLOWING rule
     declaration** — CORRECTED, and the error owned honestly (not
     blamed on the SV layout): the design-lock's earlier "PRECEDING /
     trailing-convention" reading was a **discipline miss** — I
     inferred direction from visual adjacency in `systemverilog.ebnf`
     and asserted an *unverified* claim ("the trailing `@predicate`
     after `declared_let_identifier` can only bind to that preceding
     rule") as "decisive evidence" WITHOUT checking what rule followed
     it (it actually binds forward to `known_unscoped_let_identifier`;
     verified `generated/systemverilog_parser.rs:1807` keys the
     `let_name` block to `declared_let_identifier`, the FOLLOWING
     rule). The SV layout is fully consistent with the normative
     leading-annotation rule; the binding discipline
     ([[feedback_ebnf_consult_annotation_docs]]) required consulting
     the meta-grammar FIRST and I did not. Retracted pre-commit.
     Three independent confirmations of the correct rule: (a) the
     meta-grammar `grammars/ebnf.ebnf:69-75` —
     `grammar_rule := annotation_list? rule_definition -> {…,
     annotations:$1, rule:$2}` with comment "semantic annotations that
     can **precede a rule**": the `annotation_list?` is the *leading*
     part, the rule it binds is the `rule_definition` that *follows*;
     (b) the extractor `rust/src/ebnf_frontend.rs:111-156` —
     `pending_annotations` accumulates `@…` blocks and drains via
     `std::mem::take` onto the *next* rule header (comments/blanks
     skipped); (c) the smoking-gun artifact: the first regen of the
     (mis-placed) layout keyed `@predicate`→`"backreference_digits"`
     and `@emit_fact`→`"capturing_group"` in
     `generated/regex_parser.rs` (the FOLLOWING rules, not the
     intended preceding ones). Caught by a read-only generated-parser
     sanity check *before* the probe (the probe would have shown a
     silent no-op — predicate keyed where `$2` can't resolve).
     `grammars/regex.ebnf` restructured so each `@`-directive sits
     directly ABOVE its target (`@predicate`→`numeric_backreference`
     232-233; `@emit_fact`→`capture_open` 596-597) + an in-grammar
     NOTE documenting the FOLLOWING-binding rule.
  4. **Emit at group-OPEN, not group-close**, via an open-marker
     sub-rule: `capture_open = "(" !"?"` carrying
     `@emit_fact:{kind:regex_capture_group, …}`;
     `capturing_group = capture_open pattern? ")" -> {…body:$2}`
     (positions preserved: `$1`=capture_open, `$2`=pattern?,
     `$3`=")"; `body:$2` unchanged → AST shape unchanged). Effect
     directives apply post-parse of the rule they bind to
     (`ast_based_generator.rs:945-959`); binding `@emit_fact` to
     `capture_open` (whose body is just `(`) makes the fact emit right
     after the `(` is consumed = **group-open time**, so an inner
     self/forward ref like `(\1)` sees group 1's fact and stays a
     backref (NO regression of currently-correct self-backrefs).
  5. **Capturing-vs-non-capturing precision via `!"?"`** — the
     negative lookahead makes `capture_open` fail *immediately* on
     `(?:…)`/`(?=…)`/`(?(…)…)`/`(?>…)` etc. **before** the
     side-effecting `@emit_fact`, preventing a committed-fact **leak**
     (`capturing_group` has no transaction wrapper of its own, so a
     fact emitted by `capture_open` would otherwise persist even when
     `capturing_group` later fails over a non-capturing form and the
     PEG falls to `noncapturing_group`). This does not change the
     recognized language (a `(` followed by `?` was never a
     `capturing_group`).
  6. **PCRE2 numbers named capturing groups too** — `named_group`
     (`(?<n>…)`/`(?'n'…)`) and `python_named_group` (`(?P<n>…)`) must
     also `@emit_fact regex_capture_group` (via their own open-marker
     sub-rules) so `(?<n>x)\1` keeps `\1` a backref. Non-capturing /
     lookaround / conditional / atomic groups must NOT emit.
  Remaining empirical question (settle via the regen+probe table, not
  by reasoning): whether a `regex_capture_group` fact emitted by a
  successful `capture_open` survives an **outer** PEG backtrack (e.g.
  a quantifier/alternation retry that discards an already-parsed
  capturing group) — the full-family + regression probe table
  (`(?:abc)\1` → octal; `(\1)` → backref; `(a)\1` → backref;
  `(a)(b)\2` → backref; `a(?:b)*\1`-style retry) is the arbiter.
  Copy the proven `systemverilog.ebnf` `@emit_fact`/`@predicate`
  idiom exactly; consult `grammars/return_annotation.ebnf` +
  `docs/RETURN_ANNOTATIONS_REFERENCE.md` +
  `docs/reference/PGEN_ANNOTATION_NORMATIVE_SPEC.md` per the binding
  consult-annotation-docs discipline before any `@`/`->` edit (done).
- Exact PCRE2 octal-qualification + `\8`/`\9`-no-group fallback for
  the non-backref case — `pcre2pattern(3)` rule transcribed; PCRE2
  testoutput cross-checked in `.2`.
- **`.2` FEASIBILITY CONSTRAINT (verified `semantic_runtime.rs:513-606`):**
  PGEN's current `@predicate` vocabulary
  (`has_fact`/`lacks_fact`/`has_fact_in_current_scope`/
  `has_fact_attribute`/`fact_attribute_equals`/
  `lacks_fact_attribute_equals`/`scope_depth_at_least`/
