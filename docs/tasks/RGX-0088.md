## Goal

`PGEN-RGX-0087-FIX2-0003` (rel 1.1.80) made the bare `\ddd` octal
`>0o377` an **unconditional grammar-level parse-time hard-reject**
(split `octal_escape_short_payload` + an octal-digit guard on
`class_simple_escape`). That is PCRE2-faithful for **8-bit non-UTF**
(error 151) but **WRONG under UTF / 16-/32-bit**, where a bare octal
escape is valid up to the Unicode codepoint max — PCRE2 10.47
**ACCEPTs** `\777` (U+01FF), `\400`, `[\666]`, `\6666666666` under
`,utf`. PGEN's parser is **mode-agnostic** (it never sees
`utf`/8-bit/width at parse time), so a blanket parse-time
`>0o377` reject is the wrong locus. Restore mode-agnostic octal-atom
emission; the **mode-aware consumer** range-checks by width
(the report's prescribed resolution; `feedback_ast_pipeline_parser
_agnostic`: mode is parser-context the grammar lacks).

## Authoritative PCRE2 10.47 oracle (independent — `feedback_corpus_expected_from_spec_not_fix` / `feedback_report_expected_verify_against_oracle`)

| input | 8-bit (default) | `,utf` | parser (mode-agnostic) target |
| --- | --- | --- | --- |
| `\377` `[\377]` | ACCEPT | ACCEPT | emit octal atom (ACCEPT) |
| `\400` `\777` `\666` `\6666666666` `[\666]` `[\777]` | **REJECT** (err 151) | **ACCEPT** | emit octal atom (ACCEPT) — range is consumer/mode's call |

⇒ A mode-agnostic parser cannot decide; it must emit the octal atom
for ANY `\ddd`. The 8-bit `>0o377` rejection is the mode-aware
consumer's responsibility (explicitly per the report; NOT a
`feedback_no_pgen_workarounds` violation — correct layering).

## Root cause (report + my `5fd20609` diff + oracle — grounded; empirically re-verify in `.1`)

`PGEN-RGX-0087-FIX2-0003` edited `grammars/regex.ebnf`:
- `octal_escape_short_payload = /([0-7]{1,3})/` → `= /([0-3][0-7][0-7])/ -> $1 | /([0-7][0-7]?)/ !"0"…!"7" -> $1` (an `>0o377` triple matches neither ⇒ `octal_escape` fails ⇒ hard reject).
- `class_simple_escape` gained `!"0"…!"7"` octal-digit guards (`char:$5`→`$13`) so a class-context octal-overflow `\<octal-digit>` is not shorthand-rescued.

Both make `>0o377` octal a **parse-time** reject — unconditional, mode-blind. Over-broad: rejects `\777`-class valid under UTF.
