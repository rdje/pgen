## Goal

Fix `PGEN-RGX-0086` (downstream report
`/Users/richarddje/Documents/github/rgx/pgen-issues/PGEN-RGX-0086.yaml`,
artifact `artifacts/PGEN-RGX-0086/evidence.md`):
`rust/src/embedding_api.rs`'s `REGEX_PARSER_RELEASE_VERSION`
(`"1.1.29"`) and `REGEX_PARSER_INTEGRATION_CONTRACT_VERSION`
(`"1.1.31"`) — surfaced verbatim by `parser_embedding_api_contract()`,
the canonical downstream handoff the
`PGEN_PARSER_ISSUE_REPORTING_PROTOCOL` tells consumers to copy "not
guessed from memory" — were never bumped in lockstep with
`PGEN_RELEASED_PARSER_BUG_LEDGER.md`'s regex "Fixed in" labels. They
under-report by ~46 minors, making every downstream version record
wrong-by-construction.

## Root cause / spec

The consts were last bumped at the `REGEX-0074` era ("release
1.1.29; contract 1.1.31") and ~ten ledger releases since
(`REGEX-0079..0084`) did not touch them. The **spec** for the
correct value (per the report's `expected_behavior` and the reporting
protocol): the consts must equal the *ledger's latest regex "Fixed
in"* for the built tree. After `RGX-0085` (`REGEX-0084` row) that is
**`regex parser release 1.1.77; regex integration contract 1.1.79`**
(NOT the report's stale literal `1.1.75/1.1.77`, which was captured
against an older pin — expected derived from the spec/ledger, not the
report literal, per `feedback_corpus_expected_from_spec_not_fix`).

## Non-Goals

- Not a parser-behavior change (no parse-result/AST/schema change; no
  new parser release — the consts are being corrected to *reflect*
  the parser the tree already embodies).
- Not an RGX-side workaround (RGX docs already state the
  ledger-authoritative version + a "known-stale const" annotation;
  the fix is PGEN syncing its own consts).

## Acceptance Criteria

- `REGEX_PARSER_RELEASE_VERSION = "1.1.77"`,
  `REGEX_PARSER_INTEGRATION_CONTRACT_VERSION = "1.1.79"` — exactly the
  ledger's latest regex "Fixed in"; `parser_embedding_api_contract()`
  now reports them.
- A **drift gate** (the report's explicit recommendation): a test
  that parses `PGEN_RELEASED_PARSER_BUG_LEDGER.md`, extracts the
  highest regex "Fixed in" release/contract, and asserts the consts
  equal it — so future ledger rows that forget the const bump fail
  deterministically. Spec-derived oracle (reads the ledger), not a
  hardcoded literal.
- No regression: `embedding_api` + lib + `ebnf_dual_run` suites green;
  regex parser book + mdBook gates green.
- Lockstep (binding, same-commit): ledger row for `PGEN-RGX-0086`
  (metadata-integrity; fixed by const sync + drift gate; no parser
  release bump); the `REGEX-0084` row's "deliberately untouched /
  fixed next" note + `parser-families.md`'s "fixed next" note updated
  to "fixed"; CHANGES/LIVE/memory. Then `SV-EXH-PROOF.3.1` resumes.
