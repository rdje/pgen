## Goal

Fix `PGEN-RGX-0085` (downstream report
`/Users/richarddje/Documents/github/rgx/pgen-issues/PGEN-RGX-0085.yaml`,
artifacts `artifacts/PGEN-RGX-0085/`): `pgen::embedding_api`'s regex
parse + AST-dump path imposes no effective nesting/recursion ceiling,
so a deeply nested pattern (`(((…(a)*…)*)*`) exhausts the thread
stack and HARD-ABORTS the host process (`EXC_BAD_ACCESS` on the stack
guard page → SIGABRT, uncatchable) instead of returning a recoverable
`ParseDiagnostic`. Two manifestations of the one defect: (A) recursive
`serde_json` serialization of the deep AST `Value` tree; (B) the
recursive-descent parse itself.

## Root cause (PINNED — verified from code, not assumed)

The recursion ceiling **exists and is mechanically correct**:
`generated/regex_parser.rs` constructs `RecursionGuard::new(4096usize)`;
every recursive rule calls `recursion_guard.check_cycle(...)` and the
codegen (`ast_based_generator.rs:1750-1758`) emits
`CycleType::MutualRecursive { depth } if depth >= #recursion_guard_max_depth
=> return Err(ParseError::RecursionDepthExceeded { position, depth })`
— a clean, recoverable error. The defect is purely that the bound is
unsafe relative to the stack the parser actually runs on:

- `GENERATED_RECURSION_GUARD_MAX_DEPTH = 4096` *parse-stack frames*
  ≈ 512 regex nesting levels (~8 recursive rule frames per `(`
  level). Debug per-level stack ≈ 128 KB (report: ~16 levels
  overflow a 2 MiB libtest stack). The dedicated regex worker stack
  is `GENERATED_REGEX_WORKER_STACK_BYTES` (64 MiB) → holds only
  ~512 nesting in debug — *right at / under* the 4096-frame ceiling,
  so the OS guard page faults (SIGABRT, uncatchable even though the
  worker job is `catch_unwind`-wrapped) **before** the clean ceiling
  fires.
- `embedding_api.rs` `run_generated_regex_on_dedicated_stack` runs
  nesting ≤ `GENERATED_REGEX_INLINE_DEPTH_THRESHOLD = 16` **inline
  on the caller's stack**; the in-code comment "16-deep … well under
  1 MB in debug" is empirically false (the report's
  `compile_patterns_of_increasing_complexity` SIGABRTs at ≈16 on a
  libtest worker) → manifestation B.
- `parse_generated_regex_ast_json` wraps *both* `parse_full_regex`
  and `serde_json::to_value(parsed)` inside the dedicated-stack
  closure, so for nesting > 16 the deep parse **and** the deep
  serialize share the 64 MiB worker and jointly exceed it at
  ~500 nesting → manifestation A (serde_stacker only protects
  *deserialize*, never this serialize).

## Non-Goals

- Not lowering the **global** `GENERATED_RECURSION_GUARD_MAX_DEPTH`
  (would risk legitimate deep SV/VHDL recursion — a cross-parser
  regression). The fix is scoped to the regex embedding-API
  integration boundary; the engine recursion guard is left untouched
  → provably zero SV/VHDL impact.
- Not an RGX-side workaround (doctrine: parsing/robustness issues are
  fixed in PGEN, not hacked around downstream;
  `feedback_no_pgen_workarounds`). RGX's pre-PGEN ceiling becomes
  belt-and-suspenders once PGEN owns the authoritative guard.
- Not changing the parser's correct behavior on within-ceiling input
  (control artifact `pgen_parse_outcome_control.json`: the parser is
  correct; only the ceiling is missing).

## Acceptance Criteria

- An embedding-API **configurable paren-nesting ceiling**
  (`REGEX_MAX_NESTING_DEPTH`, default = 250 — exact PCRE2
  `PCRE2_CONFIG_PARENSLIMIT` / Rust `regex` `nest_limit` parity, the
  report's own reference) is checked **before** the regex parser is
  invoked. Over-nested input → a clean `ParseDiagnostic`
  (`E_PARSE_FAILURE`) whose `byte_offset`/`line`/`column` point at
  the `(` that crossed the limit. The parser never recurses on
  over-nested input ⇒ a process abort is structurally impossible.
- `GENERATED_REGEX_INLINE_DEPTH_THRESHOLD` lowered so no
  potentially-stack-overflowing depth runs inline on an unknown
  caller stack (bench corpus max depth is 4 → keep the fast path
  only for trivially-safe shallow nesting; deeper → the 64 MiB
  worker). Stale comment corrected.
- Report verification gates, run on a 2 MiB-stack thread (so a
  guardless build fails deterministically): depth-200k and depth-2000
  → clean `Err`/Failure (NOT abort); depth-64 → Success; depth-250 →
  Success; depth-251 → clean Failure with location at the 251st `(`;
  `cargo test -p rgx-core --test stress_tests
  compile_patterns_of_increasing_complexity` shape (depths 1..=30) →
  all Success; the exact `repro_input.txt` (200k) and
  `repro_input_small.txt` (5k) → clean Failure.
- Cross-parser no-regression: SV/VHDL/regex full test suites green
  (the fix touches only the regex embedding path; the global engine
  recursion guard is unchanged — assert this by diff inspection).
- Released-parser bug remediated end to end: `REGEX-00NN` ledger row
  (downstream `PGEN-RGX-0085`), regex release/contract bump per the
  per-family convention, embedding-API contract exposes the
  configurable limit for downstream discoverability, book + contract
  + shape-contract (unchanged — no AST shape change) same-commit
  lockstep, CHANGES/DEV/LIVE/memory. No push (pacing).
