## Goal

Fix `PGEN-RGX-0087` (downstream report
`/Users/richarddje/Documents/github/rgx/pgen-issues/PGEN-RGX-0087.yaml`,
artifacts `artifacts/PGEN-RGX-0087/`): the **`[89]`-leading

matching the lone leading `8`/`9` as a numeric backref and leaving
the trailing digit(s) as literal pieces — so a pattern PCRE2 rejects
is ACCEPTED.

octal-vs-backref disambiguation). **RGX-0084 stays FIXED + CLOSED**

## Root cause (report-stated + grammar-grounded; to be EMPIRICALLY VERIFIED in `.1` before the fix)

Current post-0084 `grammars/regex.ebnf`:

```
backreference = numeric_backreference         -> $1   # :195 GATED multi-digit
              | numeric_backreference_single   -> $1   # :196 UNGATED single 1-9
              | "\\k" … | "\\g" … (subroutine/named)
@predicate: { name: fact_count_at_least, args: [regex_capture_group, $index], phase: post }
numeric_backreference        = "\\" backreference_digits        -> {type:"backreference",kind:"numeric",index:$2}  # :247 (GATED)
numeric_backreference_single = "\\" backreference_digits_single -> {type:"backreference",kind:"numeric",index:$2}  # :253 (UNGATED)
# backreference_digits = /([1-9][0-9]+)/   (multi, N≥10)
# backreference_digits_single = /([1-9])/  (single 1-9, ALWAYS backref — PCRE2 N<10 rule, RGX-0084 Non-Goal)
```

For `((((((((x))))))))\81` (8 groups): the GATED `numeric_backreference`
matches the longest run `81` (N=81); `@predicate
fact_count_at_least(regex_capture_group, 81)` is FALSE (8 < 81) → that
branch backtracks. The PEG then tries `numeric_backreference_single`
(`/([1-9])/`, UNGATED) which matches the lone `8` → emits `\8` backref
index 8 (group 8 exists ⇒ a *valid* single backref), leaving `1` a
literal piece. PCRE2 reads the **longest** run `81`: not a valid full
backref (8 groups < 81), and `8` is **not an octal digit** so no
octal fallback ⇒ PCRE2 **compile error**. The grammar has no rule
expressing "an `[89]`-leading multi-digit escape that is not a valid
*full* backref is a hard error (not a degrade to a shorter run)" —
PCRE2's longest-run semantics. (`\10` with octal digits is unaffected:
1,0 are octal so RGX-0084's octal fallback `octal_escape` catches it
— that path stays correct.)

The divergence is precisely: **enough groups exist that the leading
`\8`/`\9` split is itself a *valid* backref** ⇒ PGEN accepts where
PCRE2 rejects. `\91` w/ 8 groups already REJECTs (`\9` = missing
group 9); `(x)\81`/`\8`/`\80` already REJECT (`\8` = missing group).

Candidate fix locus (verify in `.1`): make
`numeric_backreference_single` match the single digit **only when it
is the entire digit run** (single digit NOT followed by another
digit — a negative-lookahead `!/[0-9]/`, the proven RGX-0079
negative-lookahead-guard idiom). Then `\8N`/`\9N` multi-digit runs
that fail the GATED branch fall through to the octal/literal/error
path: octal-digit runs (`\10`) → `octal_escape` (RGX-0084, stays
correct); `[89]`-leading non-octal runs (`\81`) → no octal match,
no single-digit match ⇒ clean reject (PCRE2-faithful). Single `\8`
followed by non-digit still matches single (always-backref, N<10 —
RGX-0084 Non-Goal preserved). VERIFY this is the minimal correct
shape against the full PCRE2 family table; design TBD until `.1`
empirically confirms (this is a regression from my own RGX-0084 fix
⇒ extra rigor: no assumed fix, probe the whole matrix before/after).
